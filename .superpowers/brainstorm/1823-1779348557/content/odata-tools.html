OData Tool Design

Section 2 of 4 — How the agent interacts with SuccessFactors

The Problem with "One Tool Per Entity"

SuccessFactors has 1000+ entity types in its OData metadata. Defining one tool per entity would overflow the LLM's context window and make the system unmaintainable.

Instead, we define 6 general-purpose tools that cover all entities dynamically.

The 6 OData Tools

sf_list_entities()
Returns all available entity sets from $metadata. The agent calls this first to discover what's available when it's unsure.
sf_get_schema(entity_name)
Returns field names, types, and key properties for a specific entity. Used before querying to know what fields to select or filter on.
sf_query(entity, filter?, select?,
  top?, skip?, expand?, orderby?)
GET request with full OData query parameter support. Handles pagination automatically via top + skip.
sf_create(entity_name, data)
POST to create a new record. Returns the created entity with its generated key.
sf_update(entity_name, key, data)
PATCH to update specific fields on an existing record. Key can be simple or composite (e.g., userId='U001',startDate=datetime'...').
sf_delete(entity_name, key)
DELETE a record by key. Agent will ask for confirmation before calling this in most workflows.

Example Agent Reasoning Flow

User: "Show me all employees in the Engineering department"
───────────────────────────────────────────────
🤔 Think: I need employee data. Let me check the schema first.
→ sf_get_schema("PerPerson")
🤔 Think: Found userId, firstName, lastName. Department is on EmpJob.
→ sf_query("EmpJob", filter="department eq 'Engineering'",
          select="userId,jobTitle", expand="empInfo")
✅ Returns: 12 employees with names, titles
🤔 Think: Got results. Format for user.
Response: "Found 12 employees in Engineering: ..."
