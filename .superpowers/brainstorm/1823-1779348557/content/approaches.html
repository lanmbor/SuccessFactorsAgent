Three Approaches for SF Agent

Pick the architecture that fits your priorities

A

LangChain + Chainlit

Stack: LangChain · Chainlit UI · LangChain SF tools

LangChain handles LLM switching and agent logic. Chainlit gives you a polished browser chat UI with zero frontend work. SF OData calls are wrapped as LangChain tools.

Pros

  • Fastest to build a working chat UI
  • LangChain supports 50+ LLM providers out of the box
  • Large community, many examples

Cons

  • LangChain abstractions can be hard to debug
  • Chainlit couples your UI to the framework
  • Heavy dependency tree
B

LiteLLM + Chainlit ⭐ Recommended

Stack: LiteLLM · Chainlit UI · FastAPI · Custom SF OData client

LiteLLM provides a lightweight OpenAI-compatible interface to 100+ LLM providers. Chainlit handles the browser chat UI. A small set of general-purpose OData tools (query, create, update, delete, schema-lookup) let the agent work with any SF entity without tool explosion.

Pros

  • Lightweight — no heavy framework lock-in
  • LiteLLM is battle-tested for multi-provider support
  • Beautiful chat UI from Chainlit, no frontend code
  • General OData tools scale to all SF entities
  • Easy to add Telegram / WhatsApp bots alongside

Cons

  • Slightly more glue code than full LangChain
C

LiteLLM + Custom FastAPI UI

Stack: LiteLLM · FastAPI · WebSocket · Plain HTML/JS chat UI

Fully custom — no UI framework dependencies. FastAPI serves both the chat WebSocket and the REST API for Telegram/WhatsApp webhooks. Maximum flexibility, nothing hidden.

Pros

  • Full control over every layer
  • No UI framework constraints
  • Single FastAPI service handles web + bots

Cons

  • More code to write for the chat UI
  • No built-in chat history, auth, or file handling
  • Longer time to a working prototype
